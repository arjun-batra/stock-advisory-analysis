# Stock Advisory Agent — Requirements Document (v1)

**Owner:** Arjun (solo build reference)
**Status:** Decisions closed (Section 8) — ready for build

---

## 1. Problem Statement

Manual stock-checking is inconsistent and emotion-driven. The goal is a system that applies the same disciplined judgment every time, on a personal watchlist, without requiring daily manual review.

## 2. Goals & Success Criteria

**Primary goal:** Catch a real, actionable signal on a held or watched stock that would otherwise have been missed by manual checking.

**Success criteria:** Within 3 months, at least one verdict the system surfaced is later validated as correct and would not have been caught manually. This requires the system to log calls — see Section 5.6 (resolved: logging is in for v1, since it's the only way to prove this criterion).

## 3. Scope

### In scope
- Single-user personal advisory tool (Arjun only, no multi-user/shared access)
- Watchlist of 5–15 tickers, US + Canadian (TSX) listed stocks and ETFs
- User-maintained holdings data: shares owned + cost basis per position
- AI-driven discovery of new candidates beyond the explicit watchlist, no fixed screening criteria
- Hourly threshold-triggered checks (not continuous/streaming)
- Buy / Sell / Hold verdict + one-line rationale per alert
- Push notification delivery
- AI judgment grounded in price/volume, news, and fundamentals — no fixed rule set, no fixed investment style

### Out of scope (explicit)
- Trade execution or order placement of any kind
- Brokerage account integration or read access — holdings are entered manually
- Options, crypto, derivatives, or any asset class beyond stocks/ETFs
- Multi-user or shared/team access
- Licensed financial advice — this is a personal informational tool, not a registered advisory service

## 4. Users

One user: Arjun. No external users, no handoff to another team — this doc exists as a solo build reference, not a spec for a contractor.

## 5. Functional Requirements

### 5.1 Watchlist & Holdings
- **FR1** — Maintain a list of 5–15 tickers, US and TSX stocks/ETFs.
- **FR2** — For held positions, record shares owned and cost basis; this personalizes the verdict (e.g., gain/loss context relative to entry).
- **FR3** — Tickers can be watch-only (no position held, no cost basis required).

### 5.2 Candidate Discovery
- **FR4** — Periodically scan beyond the explicit watchlist for AI-flagged candidates (US + TSX, stocks/ETFs). No fixed screening rules — the model decides relevance. Runs on a separate daily cadence, decoupled from the hourly watchlist loop (resolved in Section 8 — discovery isn't time-sensitive, and decoupling roughly halves AI call volume vs. running both scans hourly).
- **FR5** — Discovered candidates are clearly labeled as "new candidate" vs. "watchlist update" when delivered.

### 5.3 Monitoring & Triggering
- **FR6** — Checks run on an hourly cadence. No fixed daily/weekly digest format.
- **FR7** — An alert fires only when the AI's analysis crosses into an actionable call (a verdict or a change in verdict) — not on every check cycle.
- **FR8** — Alerts fire on verdict *change* only, with a 24-hour cooldown per ticker even if the verdict flips back and forth within that window — prevents spam on noisy days.

### 5.4 AI Analysis
- **FR9** — Verdicts are generated by AI judgment from price/volume data, recent news, and fundamentals — not fixed deterministic rules.
- **FR10** — No fixed investment style or horizon is assumed; the model weighs each call per stock's own context.
- **FR11** — For held positions, reasoning incorporates cost basis and position size (e.g., flags how a call relates to current gain/loss).

### 5.5 Alerting & Delivery
- **FR12** — Alerts delivered via push notification (e.g., ntfy.sh or Pushover) rather than SMS — removes the need for any SMS provider/Twilio account, and push naturally supports a tap-through link.
- **FR13** — Alert format: Buy/Sell/Hold + one-line rationale in the notification body. No long-form reasoning inline.
- **FR14** — Each notification links to a simple page showing the full reasoning behind that call, pulled directly from the log in 5.6. Natural fit for push (tap-through), avoids building two-way SMS infra.

### 5.6 Track Record
- **FR15** — Log each call: ticker, verdict, timestamp, key data points used.
- **FR16** — Logging is confirmed in for v1 — it's the only way to validate Section 2's success criterion. Kept minimal: no accuracy dashboard or analytics layer in v1.

## 6. Non-Functional Requirements

- **NFR1 — Cost:** Target $0–15/month. Hourly polling against free-tier data APIs keeps this realistic; push notification services (ntfy.sh is free, Pushover is a small one-time fee per platform) remove the per-message SMS cost entirely.
- **NFR2 — Reliability:** Scheduled checks should run dependably; a missed run shouldn't fail silently — some minimal "last run" visibility is needed.
- **NFR3 — Disclaimer:** Every alert is informational, not licensed financial advice. No regulatory registration is implied or required for personal use.
- **NFR4 — Data freshness:** Hourly cadence means up to ~60 minutes of lag is acceptable. This system is not suited for intraday/fast-moving trade timing — that was explicitly traded away for cost/simplicity (Section 8 history).

## 7. Data Sources

- **Price/volume:** Yahoo Finance unofficial API — covers both US tickers and TSX (`.TO` suffix), free. Confirmed as the v1 source.
- **News:** free headline/news feed (specific vendor TBD at build time — low-risk choice, not blocking)
- **Fundamentals:** free-tier basic financials/earnings data (same source as price/volume where possible)
- **Known risk:** Yahoo Finance's API is unofficial — no SLA, no guarantee it stays available or that TSX fundamentals data is complete. This wasn't spot-checked against your actual tickers before committing; worth a quick sanity check on day one of the build, not as a blocker but as an early smoke test.

## 8. Decisions Log (Resolved)

| # | Decision | Resolution | Rationale |
|---|---|---|---|
| 1 | Track record logging | In, minimal (no dashboard/analytics in v1) | Only way to validate the success criterion in Section 2 |
| 2 | Re-alert/dedup logic | Alert on verdict change only, 24h cooldown per ticker | Prevents spam from noisy hourly checks |
| 3 | Detail-on-request | Tap-through link from the push notification to a page reading from the log | Avoids two-way SMS infra; fits push naturally |
| 4 | Discovery scan cadence | Daily, decoupled from the hourly watchlist loop | Discovery isn't time-sensitive; roughly halves AI call volume |
| 5 | Data vendor | Yahoo Finance unofficial API (price/volume/fundamentals, US + TSX) | Free, covers both markets; unofficial-API risk noted in Section 7, not spot-checked yet |
| 6 | Notification channel | Push notification (ntfy.sh or Pushover), not SMS | Cheaper than SMS, no Twilio/CA-number dependency, supports tap-through links for detail-on-request |

## 9. Out of Scope — Explicit Confirmation

No trade execution. No brokerage integration. No options/crypto/derivatives. No multi-user access. Not a registered advisory service. These are hard boundaries for v1, not soft preferences.
